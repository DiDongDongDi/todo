# Supabase 多用户使用说明

> 笔记：不做微信改造、继续用 Supabase 时，App 能否给其他人用？  
> 整理日期：2026-06-06

## 简短结论

| 场景 | 只用 Supabase 是否可行 |
|------|------------------------|
| 每人用自己的手机/电脑，各自邮箱登录 | **基本可以** |
| 你搭一个 Supabase 项目，打包 App 分发给多人 | **可以**（云端隔离靠 RLS） |
| 同一台设备上换账号登录 | **有问题**（本地数据未隔离） |
| 不登录也能用 | **可以**（纯本地模式），但无法多设备同步 |
| 面向国内用户、要求微信登录 | **不适合**（Supabase 无原生微信） |

---

## 云端：多用户隔离已经做了

[`supabase/migrations/001_initial.sql`](../supabase/migrations/001_initial.sql) 里已有：

- `tasks.user_id` 关联 `auth.users`
- RLS 策略：`auth.uid() = user_id`

用户 A 登录后，**在 Supabase 云端**只能读写自己的任务；用户 B 看不到 A 的数据。  
[`app/lib/core/sync/sync_repository.dart`](../app/lib/core/sync/sync_repository.dart) 上传时也会带上当前登录用户的 `user_id`。

因此：**一个 Supabase 项目 + 多人各自邮箱登录**，云端数据隔离是成立的。

---

## 本地：隔离还没做（主要风险）

本地仍用**全局**存储键 `todo_tasks_v1`（见 [`app/lib/core/repositories/task_repository.dart`](../app/lib/core/repositories/task_repository.dart)），不区分用户：

1. 用户 A 在本机创建了任务
2. 退出，用户 B 登录
3. B 可能仍能看到 A 的本地任务；同步时还可能把数据搅在一起

**每人固定一台设备、从不换号**时，这个问题不明显。  
**共用平板、或同一 App 内切换账号**时，就有隐私和数据错乱风险。

---

## 「给别人用」的两种理解

### 1. 你当开发者，搭好后端，发 App 给朋友/用户

流程可以是：

1. 你在 Supabase 建一个项目，跑迁移（见 [README Supabase 配置](../README.md#supabase-配置多设备同步)）
2. 把 `url` + `anonKey` 写进发布包（或 `supabase_config.dart`）
3. 用户安装 App → 进「账号与同步」→ 邮箱魔法链接登录
4. 首次收邮件点链接即**自动注册**，无需单独注册页

这样**可以给其他人用**，且云端每人数据独立。

注意：

- **邮箱魔法链接**在手机上体验一般（要去邮件里点链接）
- 国内用户可能遇到邮件延迟 / 进垃圾箱
- App **不强制登录**：没配 Supabase 或未登录时，仍可纯本地用（见 README「不配置 Supabase 也能用」）

### 2. 把源码交给别人，各自搭 Supabase

也能用，但**不是「拿来就能用」**：每人要自己建项目、跑 SQL、填密钥，更像开发者自用，不适合普通用户。

---

## 与「微信 + 必须登录」方案的对比

| 维度 | 现状（Supabase） | 计划（微信改造，见 [微信登录与数据隔离](../.cursor/plans/微信登录与数据隔离_c36260d3.plan.md)） |
|------|------------------|---------------------------------------------------------------------------------------------|
| 登录方式 | 邮箱魔法链接 | 微信 OAuth |
| 是否必须登录 | 否，可离线本地用 | 是，未登录进不了主界面 |
| 云端隔离 | 已有 RLS | 需自建 Python API + DB 约束 |
| 本地隔离 | 未做 | 按用户独立 SQLite |
| 国内用户 | 邮箱体验一般 | 微信更自然 |
| 运维 | Supabase 托管 | 自建 FastAPI + PostgreSQL |

---

## 实际建议

- **小范围试用、每人一台设备、能接受邮箱登录** → **不必立刻做微信改造**，配好 Supabase 即可给别人用。
- **正式对外发布、国内用户、或同一设备多账号** → 至少需要补**本地按用户隔离**；若还要微信登录，才需要走微信 + 自建后端方案。

### 若继续 Supabase，最小补强清单

1. **本地按用户隔离** — 登录后使用 `users/{userId}/` 下的独立存储（Drift SQLite 或带 userId 前缀的 key），登出时切换/关闭当前库
2. **换号登出** — 退出登录时清空内存中的 Provider 状态，避免旧用户数据残留
3. **（可选）强制登录** — 若对外分发且需保证数据都进云端，可加路由守卫，未登录不进主 Tab
